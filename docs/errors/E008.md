# E008 — SQLite monitoring database is full or not writable

**Severity:** blocking. Collectors run but can't persist their data.

## What this means

The SQLite monitoring database at `monitoring_db.path` (default
`data/mysql_monitor.db`) can't be written to. Either:

- The filesystem is out of space
- The path is read-only
- The process user doesn't own the file / directory
- The configured `max_size_mb` limit is aggressively shrinking retention
  below what the workload produces

## How to verify

```bash
# Space
df -h $(dirname /path/to/data/mysql_monitor.db)

# Permissions
ls -la /path/to/data/

# Current DB size
ls -lh /path/to/data/mysql_monitor.db
```

## How to fix

**Out of disk:** free space, or move the monitoring DB to a larger
volume by changing `monitoring_db.path`.

**Permissions (Docker):** the image runs as uid `seeql`. Make sure the
host volume mount is writable by that user:

```bash
# On the host
chown -R 1000:1000 /opt/seeql/data  # 1000 is the seeql uid inside the image
```

Or mount a Docker volume instead of a host bind-mount:

```yaml
volumes:
  - seeql-data:/app/data   # Docker-managed volume, correct perms automatic
```

**Retention too aggressive:** increase `monitoring_db.max_size_mb` in
`settings.local.yaml` (default 5000 MB ≈ 5 GB). The auto-shrink only
kicks in when this is exceeded.

# E006 — MySQL connection timeout

**Severity:** blocking. No connection = no metrics.

## What this means

SeeQL opened a TCP connection attempt to `<PROD_DB_HOST>:<PROD_DB_PORT>`
and didn't get a handshake within `production_db.connect_timeout`
(default 10 seconds). Usually a networking problem, not a MySQL problem.

## When you'll see it

- At startup, before any collector query runs
- From `seeql check` / `seeql doctor`

## How to verify

```bash
# Can this machine even reach the MySQL host?
nc -vz $PROD_DB_HOST 3306

# If nc fails: DNS resolution?
getent hosts $PROD_DB_HOST

# From inside the container, if you're in Docker:
docker exec -it seeql sh -c "nc -vz $PROD_DB_HOST 3306"
```

## How to fix

**GCP Cloud SQL (private IP):** confirm the SeeQL host is in the same
VPC as the Cloud SQL instance, and that VPC peering is established.
Cloud SQL → Connections → Private IP shows the peering status.

**GCP Cloud SQL (public IP):** authorize the SeeQL host's outbound IP
in the Cloud SQL instance's Authorized Networks. For dynamic IPs, run
[Cloud SQL Auth Proxy](https://cloud.google.com/sql/docs/mysql/sql-proxy)
alongside SeeQL.

**AWS RDS:** check the security group attached to the RDS instance
allows inbound from the SeeQL host's security group / IP.

**Self-hosted:** check MySQL's `bind-address` (default `127.0.0.1`
blocks external) and any firewall rules.

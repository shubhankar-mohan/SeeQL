# E002 — `performance_schema` is disabled

**Severity:** blocking. Most collectors need `performance_schema`.

## What this means

MySQL's `performance_schema` is off. SeeQL relies on it for query
digests, lock waits, wait events, execution stages, table IO, and
several other collectors. With it off, the agent is effectively blind.

## When you'll see it

- `seeql doctor` (preflight)
- First medium-loop cycle — collectors fail with SQL errors pointing at
  missing `performance_schema.*` tables

## How to verify

```sql
SHOW VARIABLES LIKE 'performance_schema';
-- performance_schema | ON  (want this)
-- performance_schema | OFF (E002)
```

## How to fix

**Cloud SQL (GCP):** Set the `performance_schema=on` flag in the
instance parameters (console or `gcloud sql instances patch`). The
instance reboots.

**RDS / Aurora (AWS):** Edit the parameter group, set
`performance_schema=1`, and reboot the instance.

**Self-hosted MySQL:** Add to `my.cnf` under `[mysqld]`:

```ini
[mysqld]
performance_schema=ON
```

Then restart the server. On a busy production host, enabling
`performance_schema` consumes ~200 MB of memory and a small amount of
CPU — negligible on modern hardware.

## Running in limited mode

If you can't enable `performance_schema` right now, SeeQL will still
collect:

- Global status + global variables (from `SHOW GLOBAL STATUS`)
- Buffer pool stats (from `information_schema.INNODB_BUFFER_POOL_STATS`)
- Schema fingerprints and DDL changes (from `information_schema`)

You lose query digest, wait events, lock analysis, and execution stages.
